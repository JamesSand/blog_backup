Cache

Cache 类型

全相连：

简单来说是将所有 cache 表项或在一起比较。缺点是需要一个超级大的或门。好处是，任何来自内存的数据可以写入到 cache 的任何位置。

由于一个 cache 保存 4 个字节的数据，所以地址最低 2 位叫 cache offset 不用管，所以 cache index 是 30 位。

直接映射：

现在的情况是，一个内存的数据只能写入到唯一一个 cache 的位置。

因此现在的地址要分为三个部分 cache tag, cache index, cache offset

cache index 长度 = log2 cache 的表项数量

组相连：

可以理解为是多路的直接映射。简单来说，由若干个相同的 cache 来维护直接映射。CT, CI, CO 和上边的介绍一样。

Write 策略

写直达 和 拖后写：write through and write back

写直达是写入的时候直接写到内存。拖后写是写入的时候是写入到 cache

write allocate：

在 write 的时候，我们实际上还需要考虑是否将当前 write 的地方放到 cache 里边，如果 write allocate，就要把数据放到 cache 当中。

缺失原因

必然缺失：冷启动问题

容量缺失：cache 大小太小

冲突缺失：多个内存映射到同一个 cache 块。后来的把先来的挤掉了。再访问先来的就会发生冲突缺失。

无效缺失：其他进程修改了数据造成了无效。

替换策略

最近最少使用 LRU

先进先出 FIFO

随机替换 RAND

一致性保证策略

对于每一个 cache 表项，有以下几种状态

M 修改态，数据被修改过

E 独占态，只有这里的一个副本

S 共享态，可能有其他副本

I 无效态，被别人修改过了

当本地的 cache 处于 M 态，发生了远程读，这个时候读到的东西应该是修改之前的？读到的东西是不正确的。？

如果采用 Virtual Address 作为 cache 的地址

会有两个问题：

ambiguity：同一个 Vritual Address 对应着两个 Physical Address。

alias：一个 PA 对应着两个 VA

一种解决方法是，每次切换进程的时候，清空所有的 cache 表项。

一种解决思路是采用 VIPT 的方式 Virtual index Physical tag

只要保证 VA 中的 CI 和 CO 的长度加起来小于 VPO/PPO 的长度，就能保证两个不一样的 VA 的 CI 是一样的。

但是这同样需要 MMU 地址转换，为啥会更快呢？

局部性

时间局部性：最近被访问的信息还有可能会在未来被再次访问

空间局部性：最近被访问信息邻近的信息可能会被访问

IO

输入输出方式

程序直接控制：

  • 采用轮训 polling 通过汇编代码一直监视 uart 是否可读。
  • 成本低，效率低，严重占用 CPU 资源。

程序中断方式：

  • 外部设备发起请求，CPU 暂停当前工作，来处理外设的中断
  • 提高 CPU 的效率，可以控制多个外设

如果有很多外设，需要一个外设中断控制器 PLIC 来控制。处理中断的优先级，中断的栈

直接存储访问：DMA

  • IO 设备 和 主存 之间有直接的数据通路。这样主存的数据到外设就不用一定经过 CPU 了。
  • IO 外设需要通过中断向 CPU 请求读写主存，并报告结束读写主存。
  • IO 设备访问主存分为 独占总线 和 周期窃取 两种方式。前者是 CPU 完全让出 主存 总线的控制权，后者 顾名思义。

总线

总线是用来连接不同外设，和 CPU 通信的总线。

由于外设之间的速度相差很多，所以要根据不同的速度，分为不同的总线。

仲裁

集中仲裁 和 分布仲裁：

  • 集中仲裁相当于有一个统一的管理着来仲裁所有请求
  • 分布仲裁，没有统一的管理者，大家自己商量着来。
